# R4G1 Wire Format — RFC (DRAFT)

Status: DRAFT, Phase 0 deliverable of `docs/r4_graph_compiler_implementation_plan.md` (§5).
Implements PDF §18 (packed artifact format), §21 (key types), and Assumptions A1/A4 of the proof
model. Exact packed-record widths (PackedNode etc.) are chosen from measured graph sizes in
Phase 1 and frozen in the first numbered version of this document. Header-level width questions
are resolved (§9).

Terminology: `docs/transformerless/GLOSSARY.md`. Predecessor formats: TLA3/TLA4 (artifact),
TLS1 (store) — remain readable forever; R4G1 succeeds them.

## 1. Design rules (normative)

1. Single versioned container, magic `R4G1`. Explicit little-endian encoding, fixed integer
   widths, declared alignment, declared checksums.
2. Unknown **mandatory** sections MUST cause rejection. Unknown **optional** sections MAY be
   skipped per version policy.
3. All offsets are relative to their containing section start and validated with checked
   arithmetic before a `GraphView` is created.
4. Domain values crossing the serialization boundary use fixed-width types (`u8/u16/u32/u64/i32`),
   **never `usize`**, never platform-dependent layout.
5. The deployed view borrows immutable slices from caller-owned or memory-mapped bytes. The parser
   MUST NOT deserialize the graph into heap-resident object structures.
6. The parser rejects: out-of-range offsets, overlapping invalid ranges, non-canonical ordering,
   unsupported feature bits, invalid bytecode, and any bound declaration inconsistent with the
   actual ranges.
7. Canonical serialization is deterministic: identical graph content + format version ⇒ identical
   bytes (Gate E, semantics per decision D2).

## 2. Container layout

```
offset  size  field
0       4     magic "R4G1"
4       u16   format_version (major, minor as 2×u8)
6       u8    endianness marker (0x01 = little)
7       u8    alignment_log2 (section alignment, ≥ 3)
8       u64   total_len (bytes, includes header)
16      u32   section_count
20      u32   flags (feature bits; unknown mandatory bits ⇒ reject)
24      32B   artifact_cid (blake3 of everything after this field, zeroed during hashing)
56      32B   head_cid (blake3 of HEAD section body)
88      ..    section table, section_count entries × 16 bytes:
              { u32 section_id, u32 flags, u32 offset, u32 length }   (resolved, §9.1)
```

Section table entries are sorted by `section_id` (canonical ordering). Sections are padded to
`1 << alignment_log2`. Cache-line-sensitive sections (NODE/EDGE/ROUT/EMIT hot ranges) use 64-byte
alignment (Phase 7 may subdivide hot/cold).

Offsets and lengths are `u32`, section-relative, ceiling 4 GiB per section — ample against a
≤64 MB graph target and cheaper to read than u64 (§9.1). Remaining **TBD-w** (packed-record
widths: PackedNode, PackedEdge, RoutingOp, EmissionEntry) are frozen in **Phase 1** from measured
graph-size distributions (plan §5 Phase 1; PDF §21). Versioning gate for this freeze is
deterministic: pre-freeze artifacts MUST use `format_version.major = 0` (draft line), and stable
R4G1 starts at `major >= 1` only after widths are frozen. Readers/writers in the draft line MUST
reject any pre-freeze artifact that claims `major >= 1`.

## 3. Section inventory

| ID | Name | Mandatory | Contents |
|---|---|---|---|
| 0x01 | HEAD | yes | identities and limits (§4) |
| 0x02 | CODE | yes | token codes, rolling-state update programs, semantic-code layout |
| 0x03 | NODE | yes | packed region records (`PackedNode` array) |
| 0x04 | EDGE | yes | refinement (E_r), overlap (E_o), forward (E_f) edges + reverse index (E_b) |
| 0x05 | ROUT | yes | decision programs (DAG bytecode), prototypes, masks, shortlist tables |
| 0x06 | EMIT | yes | root token priors B(v), emission residuals ΔE, transition residuals ΔT |
| 0x07 | EXCT | opt. | exact-context residual evidence (successor of TLS1 store) |
| 0x08 | PROV | yes | provenance roots: source model κ, corpus CIDs, license refs, evidence roots |
| 0x09 | CERT | opt. | certification metadata: certificate CID(s), thresholds, metric refs |
| 0x0A | PTCH | opt. | patch-epoch header when the artifact is a patch layer (Phase 9) |
| 0x0B | SECT | opt. | per-section hash table — **reserved, future exploration** (§9.2) |

## 4. HEAD contents (normative field set, order fixed)

Identities:
- teacher model CID + pinned HF revision (40-hex), tokenizer identity CID
- compiler name + version, target architecture profile, deterministic seed
- corpus root CIDs (construction + held-out partitions, per decision D3)

Bounded-work constants (Theorem 4/9; validated against actual ranges):
- `A` max frontier width (u16), `C` max candidates per active node (u16),
- `W` signature words per region (u16), `E` max emission entries per region (u32),
- `K` token shortlist size (u16), `D` max decision-program steps (u32)

Starting defaults (freeze after Phase-2 frontier-width and emission histograms — issues #14/#15;
HEAD declares per-artifact maxima so values evolve per-compile without a format change):
`W=8` u64 words (512-bit codes), `A=32`, `C=16`, `E=64`, `K=8`, `D=64`, rolling window 8 tokens.

Graph dimensions: node_count (u32), edge_count (u32), depth_count (u8), vocab_size (u32),
signature_bytes (u16), ScoreQ semantic format declaration (i32 Q16.16 — §9.3).

Fallback policy: per-`ResolutionStatus` behavior codes (continue / widen / consult EXCT /
certified fallback / abstain) — the D4 policy, plus widening and exhaustive-fallback bounds.

Compatibility: min runtime version, feature bits actually required, epoch/patch-chain info
(parent CID, layer index, max layers) when PTCH present.

## 5. Section payload sketches

- **CODE**: token→code table (fixed-width entries), rolling-state program bytecode
  (shift/rotate/xor/add op list, validated), semantic-code word layout (W words × u64).
- **NODE**: `PackedNode[]` per PDF §21 — contiguous range fields (`child_start/len`,
  `forward_start/len`, `emission_start/len`, `prototype_word_start`, `mask_word_start`,
  `radius: u16`, `depth: u8`, `flags: u8`). Widths revisited TBD-w; ranges are section-relative.
- **EDGE**: canonical edge array `(u32 src, u32 dst, i16/i32 ScoreQ, u8 kind, …)` with stable edge
  IDs = array index; reverse index = ranges over a second array of edge IDs sorted by target
  (Theorem 7: reverse entries reference the same canonical edges; validator checks resolution).
- **ROUT**: decision DAG (node test opcodes over code words, child offsets, shortlist leaf
  ranges), prototype words, mask words, shortlist tables. Bytecode validated: opcodes in set,
  operands in range, DAG acyclic, depth ≤ HEAD.D.
- **EMIT**: root prior block B(v) + residual lists keyed by region (sparse, bounded ≤ HEAD.E per
  region), shared emission blocks by reference. Each table carries a dyadic **storage
  descriptor** `{width: i8|i16|i32, shift: i8, zero_point: i32}`; entries decode to semantic
  ScoreQ by shift+add at table-read time — no multiply (§9.3).
- **EXCT**: exact-context keys + residual entries (fixed-width values; key format open, §9.4).
- **PROV**: source/corpus/observation CIDs, evidence roots for deletion support, license
  identifiers (SPDX) for redistributable corpora (D3).
- **CERT**: certificate CID(s) + metric/threshold summary; the certificate document itself lives
  in the object store, not inline.

## 6. Validation invariants (Assumption A1 / Theorem 8)

Stage 1 — structural (bytes → section table):
1. magic, version supported, endianness marker, total_len == actual, section table in bounds,
   sorted, non-overlapping, aligned;
2. unknown mandatory section or feature bit ⇒ reject;
3. all offsets/lengths checked with `checked_*` arithmetic.

Stage 2 — semantic (sections → `GraphView`):
4. every packed range (children/forward/emission/prototype/mask) resolves within its target
   section;
5. every edge src/dst < node_count; reverse-index entries resolve to canonical edges whose target
   equals the containing range's node (Theorem 7);
6. ROUT bytecode: known opcodes, operands in range, acyclic, depth ≤ HEAD.D;
7. HEAD constants ≥ actual maxima observed in sections (frontier/candidates/emission bounds are
   honest);
8. emission/EXCT keys canonically ordered; ScoreQ storage descriptors valid (width in set, shift
   within declared bounds);
9. CID fields recompute to the artifact bytes (integrity).

Only after both stages may a `GraphView` be constructed; runtime table access then iterates
validated ranges only.

Phase mapping (normative intent):
- plan §5 Phase 0: publish this RFC draft and structural/semantic invariant set.
- plan §5 Phase 1: implement two-stage validator + typed `GraphView<'a>` and freeze packed widths.
- plan §6 obligations: this section operationalizes **A1** (validated-only `GraphView`) and **T8**
  (artifact-validation soundness via checked ranges → typed views).

## 7. Reproducibility semantics (decision D2 — working assumption)

- Certificate-bearing artifacts are produced by the **canonical deterministic compiler mode**:
  normative scalar FP path, pinned seeds, no platform BLAS — byte-identical output for identical
  pinned inputs on any platform (Gate E). Compiler parallelism is permitted under the §4.1 rules
  of the plan (ordered reductions, T-invariance).
- Local iteration may use platform-accelerated compiler paths; their outputs are validated by
  behavioral graph equivalence (PDF §15) rather than byte equality.
- The mode used is recorded in HEAD (compiler version + flags) and in CERT.

## 8. Versioning policy

- `format_version.major` bump: any mandatory-section or semantic change. Readers reject newer
  majors.
- `minor` bump: new optional sections/feature bits. Old readers skip unknown optional content.
- One active major at a time in this repository; TLA3/TLS1 readers remain as legacy fixtures.

## 9. Resolutions (2026-07-21) and remaining open questions

### Resolved

1. **Section-table entry and offset widths.** 16-byte entries `{u32 id, u32 flags, u32 offset,
   u32 length}`; u32 section-relative offsets/lengths (≤4 GiB per section vs ≤64 MB target).
   Rationale: fewer bytes read (Gate D) with ample headroom. Accepted.
2. **Checksums.** Single artifact CID + HEAD CID for now. **Future exploration (reserved section
   0x0B SECT):** a per-section hash table enabling mmap partial verification — verify or hot-swap
   one section without reading the whole artifact; candidate for a v1.x feature bit or the
   PTCH-era format. Revisit when patch layers (Phase 9) or partial-memory deployments make it
   concrete.
3. **ScoreQ.** Two-tier design. *Semantic*: one global format — i32 Q16.16 declared in HEAD
   (range/resolution ample for log-domain scores; a single declared semantics keeps A4 and
   Theorem 6 replay simple). *Storage*: per-table dyadic descriptors `{width, shift, zero_point}`
   (see EMIT, §5); decode is shift+add at table-read time — mul-free, matching the existing
   decode-on-demand pattern (`runtime.rs:134-147`, `compiler.rs:456-470`). Per-depth scales fall
   out as a special case. Per-table parameters chosen from Phase-4 residual histograms.

### Still open

4. **EXCT key format.** Leaning prefix-ordered byte keys (preserve graded prefix backoff, TLS1
   lineage) vs. fixed-width hashed keys (uniform width, cache-friendlier, cheaper witness
   replay). Decider: Phase 4 — whether EXCT keeps prefix-backoff semantics, plus measured replay
   cost (Theorem 6).
5. **PTCH layering details** (bounded lookup depth, compaction triggers, fork handling).
   Deferred to the Phase 9 RFC as planned; nothing in Phases 1–8 depends on it.
